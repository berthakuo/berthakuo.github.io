Child Scholarship Plan

Background

Child Scholarship Plan (CSP) is a tuition reimbursement benefit MIT provides to eligible employees. The purpose of this project was to digitize the existing paper enrollment process and replace the outdated benefit management system (Filemaker).

My Role

Though both employees and administrators interface with the system, my focus was on designing the UI for the administrative-side of the system.

Process

Challenge 1

Getting up to speed

Once I had a general understanding of how the benefit process worked, my next step was to make a list of the objects that needed to be represented in the system and their components, actions the user wants to take on each object, and potential states and statuses of each object.

This exercise not only reinforced my understanding of the system but helped surface things that were unclear.

Next, I took the main objects (employee, student, application) and sketched out their relationships which served as a foundation for the design. From there there I began layering on additional views (detail-view, list-view) that corresponded to tasks the user needed to accomplish.

Challenge 2

Where are the wireframes?!

When I joined the team, the project was behind schedule and there was pressure to quickly produce wireframes despite it being unclear how the new system would impact the existing business process.

Because I knew what the system objects were, I started by sketching out each object using the output from the previous exercise as a guideline. I treated each object as a stand-alone component; I didn’t know what the final configuration/arrangement of these components would be but I knew that no matter what, the component would be present in the system.

This “inside-out method” proved to be an effective way of demonstrating that progress was being made without wireframing screens or knowing how everything was going to come together.

Challenge 3

You Are Not Your User

The ideal design process starts with user research. Though I was able to create a map of the objects in the system using OOUX, without the user component to anchor the design, I was concerned that we would build/create something that was not useful or effective. The team, however, was reluctant to let us conduct user interviews at this stage of the project so we needed to be creative in how we gathered this information.

My solution was to request a copy of all the administrator’s notes entered into the original Filemaker system. My reasoning was that the notes, an optional field, would provide insight into what the user thought was important to record rather than what the system thought was important to record. The notes were a window into the administrator’s thought process; we used them in lieu of direct user observation. Because the notes only revealed what the user did and not why, I made sure to write down any questions, interpretations or assumptions on my part that I would later validate with the user.

Reviewing these notes gave me a better understanding of the different scenarios the user might encounter and the subsequent needs and pain points associated with those scenarios. It also inspired design features that were not initially articulated by the business. Finally, the ability to anchor our conversations around these notes rather than speak in generalizations or hypotheticals resulted in more productive team discussions.

Challenge 4

Making Sense of it All

While I focused on designing the admin-side of the system, my colleague was designing the employee-facing side. Though we were designing for different end users, the UIs were interconnected; a change on the employee-side could affect the admin-side and vice versa. Therefore, we had to make sure that our designs would properly accommodate for tasks that traversed both UIs.

In order to get the team thinking about the wireframes holistically, I taped printouts of each wireframe (employee and admin) onto a large wall and organized an end-to-end walk through of all the activities that occur during the life-cycle of a benefit.

I was astonished by how the team embraced the walk through.

Unlike the weekly wireframe meetings, in which a single screen was presented for feedback, the team began engaging more critically with the wireframes and started thinking about the implications that the wireframes had on the back end.

Deliverables

At the heart of the system is the “claim” which defines how much money each employee will receive and when. Below are three case studies that demonstrate how the claim’s design evolved throughout the project.

Case Study 1

Show Don't Tell

In the previous system, the benefit amount each applicant is eligible for is calculated by the administrator. The new system calculates this value automatically.

Based on the end user’s lengthy experience with administering benefits, my initial design relegated the calculation explanation to an information icon (1).

However, as the project continued, I became aware of unique cases (e.g., non-USD tuition for foreign universities) that added complexity to the benefit calculation. A tooltip, which is usually used for short descriptions, was not the ideal format to communicate this complexity.

Show, Don’t Tell

In this design, the benefit explanation is stored in a vertical drawer that is accessible via the “show calculation” (2) button. Rather than just provide a generic description of the calculation, the additional space allowed me to show each step of the calculation using actual values. By seeing every step, the user can be confident in the validity of the final value.

Case Study 2

Side-By-Side Comparison

In the initial design, the UI displayed the system-calculated benefit (SCB) as well as how that benefit would be paid-out through the year.

Issue: Though the payment amounts (3) were editable, the recommended SCB (4) was not. The team was concerned that this would confuse the user and proposed that the SCB be automatically updated if the payment amounts were changed. When that proved to be confusing, they suggested using a warning banner to explain that the SCB did not take into account manual overrides.

These solutions assumed that the problem was the SCB. However, the problem was that the UI did not provide the information needed for the user to confidently override a system calculated value.

Solution: In order to clearly communicate the consequence of updating an individual payment on the total sum, a second column tallying the manual inputted values was included alongside the system-calculated values.

By showing both the system-calculated benefit side-by-side with the benefit amount based on a manual override, the user has a clearer picture

Case Study 3

Explain the Consequences of an Action

Another use case that needed to be addressed was if a benefit recipient changed schools. Because the existing system did not allow for changes to approved applications, payments had to be manually updated in the database. The team envisioned that once a “change form” was approved in the new system, payments would be automatically updated.

Since most of the work associated with processing a change form was offloaded onto the system, there wasn’t much difference between how an administrator processed a new form vs. a change form. Therefore, the claim section of a change form was based on the new application form.

Issue: However, by approving a “change form”, the administrator was not only creating a new payment but also cancelling any previously scheduled payments. This consequence was not visible in the UI and therefore a potential source of confusion for anyone unfamiliar with the system.

Solution: This design better facilitates decision-making by indicating the direct and indirect consequences of approving a form. In the example below, color is used to indicate what will happen to each claim payment if this application is approved.

  • White (5) : This claim has already been paid and will be unaffected.
  • Red (6) : This payment will be canceled.
  • Green (7) : This payment will be added.

In addition, showing both the original and changed payments makes it clear that these claims are associated with the same benefit.
